## 1. แอปนี้คืออะไร

เว็บแอป (Next.js 14) ช่วยวางแผน **เส้นทางเดินเท้าในกรุงเทพ** ให้ปลอดภัย/ร่ม/สบาย
โซน demo: **ย่านปทุมวัน** (เช่น สยาม / สนามกีฬาแห่งชาติ → รพ.จุฬาฯ) — ใช้ **ข้อมูลจริงทั้งหมด ไม่มี mock**

ปัจจัยที่นำมาคิดคะแนนเส้นทาง: ความปลอดภัย (จุดร้องเรียน + กล้อง CCTV), ร่มเงาจากต้นไม้, พื้นที่สีเขียว/สวน, ห้องน้ำใกล้ทาง, จุดมืด/แสงสว่าง, จุดน้ำท่วม — และ **ปรับน้ำหนักตามเวลา กลางวัน/กลางคืน**

## 2. สถาปัตยกรรม & ไฟล์สำคัญ

แอปเป็น Next.js App Router — UI ฝั่ง client + API routes ฝั่ง server (กัน CORS / ซ่อน key / cache)

```
walkbkk/
├─ app/
│  ├─ page.jsx                 # ประกอบ MapView + ChatBox
│  ├─ layout.jsx
│  └─ api/
│     ├─ route/route.js        # เรียก OpenRouteService (foot-walking, 3 ทางเลือก)
│     ├─ chat/route.js         # ผู้ช่วย AI (Typhoon ThaiLLM) — ตอบอิงข้อมูลจริง
│     ├─ traffy/route.js       # จุดร้องเรียน Traffy Fondue (สด CKAN + fallback geojson)
│     ├─ floodrisk/route.js    # จุดเสี่ยงน้ำท่วม กทม. (สนน. 2566) จาก CSV
│     ├─ osm/route.js          # OSM Overpass: ต้นไม้/ห้องน้ำ/สวน/กล้อง/ทางข้าม
│     ├─ toilets/route.js      # ห้องน้ำ OSM (สำรอง)
│     └─ flood/route.js        # (สำรวจ) endpoint น้ำท่วมเรียลไทม์ กทม.
├─ components/
│  ├─ MapView.jsx              # แผนที่ Leaflet + คำนวณคะแนน + นำทาง GPS/2D + ค้นหา
│  ├─ ChatBox.jsx              # หน้าต่างแชต Typhoon
│  └─ Nav3D.jsx               # โหมดนำทาง 3D (MapLibre GL) แยกจากแผนที่หลัก
└─ public/data/
   ├─ unresolved_pathumwan.geojson   # cache จุดร้องเรียน Traffy (ข้อมูลจริง)
   └─ flood_risk.csv                 # จุดเสี่ยงน้ำท่วม กทม.
```

### แหล่งข้อมูล
- **Traffy Fondue** — คำร้องเรียนทางเท้า/ถนน/น้ำท่วม/แสงสว่าง/กีดขวาง/กล้อง (cat: `sidewalk`, `road`, `flood`, `light`, `obstruct`, `cctv_broken`) ดึงสดจาก CKAN `data.bangkok.go.th` ถ้าล่มใช้ geojson cache
- **สำนักการระบายน้ำ กทม.** — จุดเสี่ยงน้ำท่วม (cat: `flood_risk`) จาก CSV → geocode ฝั่งเบราว์เซอร์
- **OSM (Overpass + Nominatim)** — ห้องน้ำ, กล้อง CCTV, ต้นไม้, สวน, ทางม้าลาย, geocode/reverse-geocode
- **OpenRouteService** — เส้นทางเดิน 3 ทางเลือก
- **Typhoon (opentyphoon.ai)** — LLM ภาษาไทยสำหรับแชต

### สูตรคะแนน (`scoreRoutes` ใน MapView)
- **กลางวัน (07:00–18:00):** safe 0.35, ร่ม 0.30, สวน 0.20, ห้องน้ำ 0.15 — ไม่หักจุดมืด
- **กลางคืน:** safe 0.55, ร่ม 0.0, สวน 0.10, ห้องน้ำ 0.15 — จุดมืดอันตรายขึ้น (×1.5)
- คำนวณ 2 จังหวะ: **quick** (ใช้ Traffy ทันที) แล้ว **full** (เติม OSM ร่ม/สวน/ห้องน้ำเบื้องหลัง)

## 3. สิ่งที่ทำไปแล้ว (changelog หลัง demo แรก)

### ✅ แก้พิกัดแลนด์มาร์กให้ทนทาน (OSM-backed)
- เดิม รพ.จุฬาฯ ปักหมุดผิดฝั่งถนน (พิกัด hardcode `[100.5366, 13.7295]` ตกขอบพระราม 4)
- เพิ่ม `resolveLandmark()` — ดึงพิกัดจริงจาก Nominatim โดย **ยอมรับเฉพาะเมื่ออยู่ใกล้พิกัด curated <1.5 กม.** (กันคืนที่ผิด) + fallback ออฟไลน์ + cache localStorage
- แก้พิกัด รพ.จุฬาฯ เป็น `[100.5356, 13.7314]` + ใส่ `query` ค้น OSM
- เดินสายครบ: ช่องค้นหา, autocomplete, `DEF_END` ฝั่ง server

### ✅ เสียงนำทางเสถียร (ทั้ง 2D และ 3D)
- เก็บ reference ของ `SpeechSynthesisUtterance` กัน GC ตัดเสียง → แก้ "พูดไม่จบ/เป็นคำๆ"
- watchdog เรียก `resume()` ทุก 3 วิ → กันบั๊ก Chrome หยุดพูดเองหลัง ~15 วิ
- ตรวจสถานะ `speaking` ค้าง >12 วิ แล้ว reset → แก้ "3D เงียบสนิท"
- ใช้เทคนิคชุดเดียวกันทั้ง `MapView` (2D) และ `Nav3D` (3D)

### ✅ แก้เตือน "เตรียมข้ามถนน" ผิด (2D)
- เดิมเตือนทุกครั้งที่มีหมุดทางข้าม OSM ใกล้เส้นทาง ≤20 ม. → บนถนนยาวเตือนมั่ว
- ใหม่: ต้องอยู่บนเส้นทางจริง **≤10 ม. และ** มีจุดเลี้ยวจริงใกล้ๆ (±25 ม.) ถึงเตือน
- ข้อแลก: ทางข้ามกลางบล็อกที่ไม่เลี้ยวอาจไม่เตือน (ยอมรับได้ ดีกว่าเตือนมั่ว)

### ✅ Typhoon ตอบเรื่องห้องน้ำได้จริง
- `scoreRoutes` แนบ `toiletList` ต่อเส้นทาง: ชื่อ + ระยะจากต้นทาง + ระยะห่างจากเส้นทาง + **ถนน** (จาก route steps) + `pt`
- **reverse geocode** (คิว+cache) เติม **ชื่อตึก/ย่าน** → ตอบได้ว่า "อยู่ตึกไหน ถนนอะไร"
- chat API ลิสต์ห้องน้ำต่อเส้น + บังคับให้ตอบจากรายการเท่านั้น (ห้ามเดา)
- **แยก intent**: ถ้าถามเรื่องห้องน้ำ ใช้ system prompt เฉพาะ (กันสลับไปตอบรูปแบบเปรียบเทียบเส้นทาง)
- **ทำหมุดบนแผนที่ตรงกับคำตอบ**: รวมตัววาดหมุดห้องน้ำ/กล้องแบบกันซ้ำ เรียกทั้งตอนโหลด demo และตอนค้นเส้นทาง (เดิมหมุดมาจาก DEMO_BBOX ส่วนคำตอบมาจากกรอบเส้นทาง → ไม่ตรงกัน)
- threshold "ใกล้เส้นทาง" 150 → **120 ม.**

### ✅ ความแม่นยำตำแหน่งห้องน้ำ (precision fix — จุดขายของแอป)
- **บั๊กเดิม:** วัด "ห่างเส้นทาง" + "กม.จากต้นทาง" จากระยะถึง *จุดหักมุม (vertex)* ที่ใกล้สุด ไม่ใช่ระยะตั้งฉากถึงเส้นจริง → บนถนนยาว (เช่นพระราม 4) คลาดเคลื่อนมาก · ทดสอบแล้ว: ห้องน้ำที่ห่างจริง 30 ม. ถูกรายงานเป็น **293 ม.** → โดน threshold ตัดทิ้งทั้งที่อยู่ใกล้ (false negative)
- **แก้:** เพิ่ม `nearestOnRoute()` ฉายจุดตั้งฉากลง segment → ได้ `off` (ระยะตั้งฉากจริง) + `along` (ตำแหน่งตามแนวเดินจริง) + `seg` (ไว้หาชื่อถนน) แม่นยำ
- **ยุบรายการซ้ำ:** ห้องน้ำหลาย node ที่ห่างกัน ≤30 ม. (ถูก tag ซ้ำ) ยุบเหลือจุดเดียว เก็บอันที่ใกล้เส้นสุด → ลิสต์ไม่รก
- **ชื่อตำแหน่งแม่นขึ้น:** `reverseGeocode` เลือก POI/ตึก/สวนจริงก่อน เลี่ยง `suburb` (ระดับแขวง กว้างเกิน ทำให้เข้าใจผิดว่าอยู่คนละที่ เช่น "สี่พระยา/สันติภาพ")
- ทดสอบ `node` ผ่าน 5/5 (เทียบ vertex vs ตั้งฉาก, along กลางเส้น, dedup)

### ✅ "ห้องน้ำใกล้สุดข้างหน้า" ตอนนำทาง GPS (จุดขาย — เดินอยู่แล้วอยากเข้าห้องน้ำ)
- แนบ `toiletList` (along/off แม่นแล้ว) เข้า `c.nav` ทั้งโหมด GPS จริงและโหมดจำลอง
- `updateNav` หาห้องน้ำที่ **อยู่ข้างหน้าตามแนวเดิน** (along > ตำแหน่งผู้ใช้) ที่ "ถึงเร็วสุด" = ระยะข้างหน้า + ระยะเบี่ยง · ตัดอันที่ผ่านไปแล้ว / ไกล >300 ม. / เบี่ยงจากทาง >90 ม.
- **แสดงผล:** แถบ 🚻 ในแผงนำทาง บอกระยะข้างหน้า + เบี่ยงจากทาง + ชื่อตึก/ถนน
- **เสียง:** พูด "ห้องน้ำอีก X เมตรข้างหน้า" เมื่อเข้าใกล้ ≤45 ม. (พูดครั้งเดียวต่อจุด, ไม่ตัดเสียงนำทางเลี้ยว) · รองรับ TH/EN
- ทดสอบ `node` ผ่าน 5/5 (เลือกจุดถึงเร็วสุด, ข้ามจุดที่ผ่าน/ไกล/เบี่ยงเกิน)

### ✅ คำตอบความปลอดภัยปรับตามเวลา
- ส่งสถานะ `night` ให้ Typhoon
- กลางวัน: ตัดบรรทัด "จุดมืด" ออก (ไม่เกี่ยว) + สั่งห้ามอ้างความมืด
- กลางคืน: ตัด "ร่ม" ออก + เน้นแสงสว่าง/จุดมืด

### ✅ แยกชนิดน้ำท่วมให้ชัดในคำตอบ
- เดิมส่งแค่ `floodN` (คำร้องเรียน Traffy) แต่โมเดลเรียกว่า "จุดเสี่ยงน้ำท่วม" → สับสนกับชั้น กทม.
- ใหม่ส่งทั้ง `น้ำท่วม-ร้องเรียน(Traffy)` และ `เสี่ยงน้ำท่วม-กทม.` (`floodRiskN`) แยกป้าย
- สั่งโมเดล: ปัจจัยที่ **เท่ากันทุกเส้น/เป็น 0** ห้ามยกเป็นข้อได้เปรียบ

### ✅ โหมด 3D โชว์ห้องน้ำ (W) + กล้อง CCTV (C)
- เก็บ `ctx.osmToilets` / `osmCameras` (ชุดเดียวกับแผนที่หลัก) ส่งเข้า `Nav3D`
- `Nav3D` กรองเฉพาะที่อยู่ใกล้เส้นทาง (ห้องน้ำ ≤120 ม., กล้อง ≤120 ม.) แล้ววาดหมุด + popup
- fallback เป็น `route.toiletList`/`cameraList`

### ✅ แก้ UI ทับกัน
- พาเนลเส้นทางหดให้สูงไม่เกิน `calc(100vh - 500px)` เมื่อเปิดแชต + ยก z-index แชต กันโดนบัง

## 4. สิ่งที่ยังค้าง / ข้อควรรู้สำหรับงานต่อ

- **API key ใส่แล้ว** ใน `.env.local`: `ORS_API_KEY` (OpenRouteService) + `TYPHOON_API_KEY` (opentyphoon.ai) ✅ — routing และแชตใช้งานได้ · *หมายเหตุ: ถ้า deploy ขึ้น Vercel ต้องไปใส่ env เหล่านี้ใน Project Settings ด้วย (ไฟล์ `.env.local` ไม่ถูก commit)*

- **ข้อมูล OSM ห้องน้ำ/กล้องบางซอยเล็กมีน้อย** — เส้นที่ลัดเข้าตรอก/ซอยอาจไม่มีหมุด (เป็นข้อจำกัดของข้อมูล ไม่ใช่บั๊ก)
- **AR นำทางกล้องจริง** (ตาม `ROADMAP.md`) ยังไม่เริ่ม
- **กลางวัน/กลางคืน** อิงเวลาบนเครื่องผู้ใช้ (07:00–18:00 = กลางวัน) มี fallback เวลาไทยฝั่ง server

### หมายเหตุการทดสอบ/ดีพลอย
- หลังแก้โค้ด: **รีเฟรช + กดค้นหาเส้นทางใหม่** เพื่อให้ระบบคำนวณ/แนบข้อมูลรอบใหม่
- ถ้าพิกัด/ข้อมูลเก่าค้าง: ล้าง localStorage (คีย์ `fg:`, `lmpos:`, `rev:`, `osm:`) หรือ `localStorage.clear()` ใน DevTools Console
- การ verify รอบนี้ทำแบบ **syntax-check แยกส่วน** (รัน `next build` เต็มในเครื่อง sandbox ไม่ได้: `.next` ถูกล็อกจาก dev server + mount เห็นไฟล์เวอร์ชันเก่า) — ควรรัน `npm run build` บนเครื่องจริงก่อนดีพลอยเพื่อยืนยัน

## 6. 🌳 เฟสถัดไป (วางแผนไว้): ปรับ "การคำนวณร่ม" ให้สมจริง

> **สถานะปัจจุบัน:** ร่ม = สัดส่วนจุดบนเส้นทาง (สุ่มทุก 25 ม.) ที่มีต้นไม้ OSM (`natural=tree`) ภายใน 25 ม. × 100 — ใช้ปัจจัยเดียวคือต้นไม้ ไม่คิดทิศแดด/เงาตึก/หลังคา
> **เป้าหมาย:** ให้ "ร่ม" สะท้อนความจริงในเมือง = ทิศทางแดด + เงาตึก + skywalk/หลังคา + ต้นไม้ใหญ่

### ประเมินความพร้อมข้อมูล (✅ ยืนยันด้วยตัวเลขจริงแล้ว — `scripts/probe_shade_data.mjs`, bbox ปทุมวัน, 2026-06-21)
| ปัจจัย | แหล่งข้อมูล | ความพร้อม | ตัวเลขจริง (ย่าน demo) |
|---|---|---|---|
| ทิศทางแดด ☀️ | คำนวณเอง (lat/lon+วันเวลา, lib `suncalc`) | ✅ 100% ไม่ต้องพึ่งข้อมูล | คำนวณได้เลย |
| skywalk / ทางมีหลังคา 🏬 | OSM `covered=yes/arcade`, `bridge`(footway), `man_made=bridge` | ✅ พอใช้จริง (มากกว่าที่คิด) | footway `covered`=**76**, `bridge`=**115**, `man_made=bridge`=**10** (จาก footway 959 เส้น) |
